## 1. 背景与业务目标

实现患者管理的全部功能，为医疗系统提供完整的患者信息生命周期管理能力，包括患者信息的新增、查询、修改、删除与详情查看。

### 1.1 问题陈述

当前缺少统一的患者信息管理平台，导致患者数据分散、查询效率低，难以支撑医疗业务的精细化运营。

### 1.2 业务目标

| 目标 | 衡量指标 |
|------|----------|
| 提升患者信息录入效率 | 单条患者信息录入时间 ≤ 2 分钟 |
| 支持快速检索 | 列表查询响应时间 ≤ 500ms（万级数据量） |
| 保证数据完整性 | 关键字段填写完整率 ≥ 99% |
| 操作可追溯 | 所有增删改操作记录操作人与时间戳 |

## 4. 功能需求详述

### 4.1 患者信息新增

#### 4.1.1 功能描述

授权用户可通过填写表单录入新患者的基本信息，系统自动生成唯一患者编号。

#### 4.1.2 患者信息字段

| 字段名 | 字段类型 | 是否必填 | 约束说明 |
|--------|----------|----------|----------|
| 患者编号 | 字符串 | 系统自动生成 | 格式：`PAT-YYYYMMDD-NNNN`，全局唯一 |
| 姓名 | 字符串 | 是 | 2–50 个字符 |
| 性别 | 枚举 | 是 | 男 / 女 / 未知 |
| 出生日期 | 日期 | 是 | 格式：YYYY-MM-DD，不得晚于录入时的系统当前日期 |
| 身份证号 | 字符串 | 否 | 满足中国居民身份证格式，系统级唯一 |
| 手机号 | 字符串 | 是 | 11 位手机号，系统级唯一 |
| 联系地址 | 字符串 | 否 | 最多 200 个字符 |
| 紧急联系人姓名 | 字符串 | 否 | 最多 50 个字符 |
| 紧急联系人电话 | 字符串 | 否 | 11 位手机号 |
| 过敏史 | 文本 | 否 | 最多 500 个字符 |
| 备注 | 文本 | 否 | 最多 1000 个字符 |
| 创建时间 | 日期时间 | 系统自动填充 | UTC 时间，精确到秒 |
| 创建人 | 字符串 | 系统自动填充 | 当前登录用户 |

#### 4.1.3 业务规则

- 手机号与身份证号（如已填写）在系统中不得重复，重复时提示"该手机号/身份证号已存在"。
- 必填字段缺失时，表单不得提交并高亮错误字段。
- 提交成功后页面跳转到新建患者的详情页。

#### 4.1.4 异常处理

| 场景 | 系统行为 |
|------|----------|
| 手机号已存在 | 返回错误提示，不写入数据库 |
| 身份证号格式不合法 | 前端实时校验，阻止提交 |
| 网络超时 | 提示"保存失败，请重试" |

---

### 4.2 患者信息查询（分页、筛选）

#### 4.2.1 功能描述

以分页列表形式展示患者信息，支持多维度筛选，帮助用户快速定位目标患者。

#### 4.2.2 查询参数

| 参数 | 类型 | 说明 |
|------|------|------|
| 关键字 | 字符串 | 支持患者编号、姓名、手机号模糊搜索 |
| 性别 | 枚举 | 男 / 女 / 未知 |
| 出生日期范围 | 日期区间 | 开始日期 ~ 结束日期 |
| 创建时间范围 | 日期时间区间 | 开始时间 ~ 结束时间 |
| 页码 | 整数 | 默认第 1 页 |
| 每页条数 | 整数 | 默认 20，可选 10 / 20 / 50 / 100 |

#### 4.2.3 列表展示字段

患者编号、姓名、性别、年龄（由出生日期计算）、手机号、创建时间、操作（查看 / 编辑 / 删除）。

#### 4.2.4 排序

默认按创建时间降序；支持按姓名、创建时间列头点击升/降序切换。

#### 4.2.5 业务规则

- 无筛选条件时展示全部未删除患者。
- 搜索结果为空时展示"暂无数据"占位符。
- 分页信息需显示总记录数与当前页范围（例如"共 1234 条，第 1-20 条"）。

---

### 4.3 患者信息修改

#### 4.3.1 功能描述

授权用户可对已有患者信息进行编辑，修改记录自动追踪变更历史。

#### 4.3.2 可编辑字段

除**患者编号**、**创建时间**、**创建人**外，其余字段均可修改。

#### 4.3.3 业务规则

- 修改前需对当前数据进行乐观锁校验，防止并发覆盖。
- 修改成功后记录修改人与修改时间，可在详情页变更历史中查看。
- 必填字段不得置空；唯一性约束（手机号、身份证号）同新增规则一致。

#### 4.3.4 异常处理

| 场景 | 系统行为 |
|------|----------|
| 乐观锁冲突 | 提示"数据已被他人修改，请刷新后重试" |
| 患者记录不存在 | 返回 404，跳转到列表页并提示"患者不存在" |

---

### 4.4 患者信息删除

#### 4.4.1 功能描述

采用**逻辑删除**方式，标记记录为已删除状态，数据不物理移除，以支持数据审计和恢复。

#### 4.4.2 业务规则

- 删除前弹出确认对话框："确认删除患者【{姓名}】？此操作不可撤销。"
- 用户确认后执行逻辑删除，设置 `deleted_at` 与 `deleted_by` 字段。
- 删除后该患者记录不再出现在查询列表中（默认过滤已删除数据）。
- 删除成功后刷新当前列表页。
- 批量删除：支持勾选多条记录后统一删除（不超过 50 条/次）。

#### 4.4.3 权限控制

- 仅**医院管理员**与**系统管理员**具有删除权限。
- 医生/护士角色不展示删除操作入口。

---

### 4.5 患者信息详情

#### 4.5.1 功能描述

以卡片/表单形式展示患者的完整信息，包含基本信息、操作历史等。

#### 4.5.2 展示内容

**基本信息区**：患者编号、姓名、性别、年龄、出生日期、身份证号（脱敏展示）、手机号（脱敏展示）、联系地址、紧急联系人姓名与电话、过敏史、备注。

**元数据区**：创建时间、创建人、最后修改时间、最后修改人。

**变更历史区**：按时间倒序展示每次修改的字段、修改前值、修改后值、修改人、修改时间。

#### 4.5.3 敏感数据脱敏规则

| 字段 | 脱敏规则 | 示例 |
|------|----------|------|
| 身份证号 | 保留前 3 位和后 4 位，中间替换为 `*` | `110***********1234` |
| 手机号 | 保留前 3 位和后 4 位，中间替换为 `****` | `138****5678` |

## 5. 非功能需求

### 5.1 性能

| 指标 | 目标值 |
|------|--------|
| 列表查询（10 万条数据）| 响应时间 ≤ 500ms（P99） |
| 单条新增/修改 | 响应时间 ≤ 300ms（P99） |
| 单条删除 | 响应时间 ≤ 300ms（P99） |
| 详情查询 | 响应时间 ≤ 200ms（P99） |
| 系统并发用户数 | 支持 ≥ 100 并发 |

### 5.2 安全

- 所有接口需经过身份认证（JWT / Session Token）。
- 权限校验基于 RBAC 模型，未授权访问返回 403。
- 敏感字段（身份证号、手机号）在传输和存储时加密。
- 操作日志记录用户 IP、操作类型、操作时间，保留 ≥ 1 年。

### 5.3 可用性

- 系统可用性 ≥ 99.9%（月维度）。
- 支持主流浏览器：Chrome 90+、Firefox 88+、Edge 90+、Safari 14+。

### 5.4 数据一致性

- 删除操作使用事务保证原子性。
- 并发修改使用乐观锁防止数据覆盖。

## 7. 数据模型（概要）

```sql
-- 患者主表（详细设计由 role:architect 完善）
CREATE TABLE patients (
  id            BIGINT          PRIMARY KEY AUTO_INCREMENT,
  patient_no    VARCHAR(30)     NOT NULL UNIQUE,   -- PAT-YYYYMMDD-NNNN
  name          VARCHAR(50)     NOT NULL,
  gender        TINYINT         NOT NULL,           -- 0=未知 1=男 2=女
  birth_date    DATE            NOT NULL,
  id_card       VARCHAR(255),                       -- 加密存储
  phone         VARCHAR(255)    NOT NULL,           -- 加密存储
  address       VARCHAR(200),
  emergency_contact_name  VARCHAR(50),
  emergency_contact_phone VARCHAR(20),
  allergy_history TEXT,
  remark        TEXT,
  created_at    DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP,
  created_by    VARCHAR(100)    NOT NULL,
  updated_at    DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  updated_by    VARCHAR(100),
  deleted_at    DATETIME,
  deleted_by    VARCHAR(100)
);
```

## 8. 验收标准

| 编号 | 功能点 | 验收条件 |
|------|--------|----------|
| AC-01 | 患者信息新增 | 填写合法信息后点击保存，系统生成唯一编号并跳转详情页 |
| AC-02 | 患者信息新增（校验） | 手机号重复/格式错误时给出明确提示，不写入数据库 |
| AC-03 | 患者信息查询 | 按姓名/编号/手机号模糊搜索，结果准确，分页正确 |
| AC-04 | 患者信息查询（空结果） | 无匹配时显示"暂无数据" |
| AC-05 | 患者信息修改 | 修改后数据正确持久化，变更历史记录可见 |
| AC-06 | 患者信息修改（乐观锁） | 并发修改时给出冲突提示 |
| AC-07 | 患者信息删除 | 确认后逻辑删除，列表不再展示该患者 |
| AC-08 | 患者信息删除（权限） | 医生/护士角色不可见删除入口 |
| AC-09 | 患者信息详情 | 身份证号与手机号脱敏展示正确 |
| AC-10 | 敏感数据安全 | 未登录用户访问患者接口返回 401 |

# 患者管理模块 - 产品需求文档 (PRD)

> 交付角色：`role:product`
> 关联根任务：患者管理模块
> 版本：v1.0
> 最后更新：2026-04-20

---

## 1. 背景与目标

### 1.1 背景
在医疗/诊所类业务系统中，患者（Patient）是一切业务流程（挂号、就诊、开方、收费、随访等）的主体实体。当前系统缺少一个统一、可靠的患者主档管理能力，亟需落地一个**最小但完整**的患者信息管理模块，为后续业务模块提供基础数据支撑。

### 1.2 业务目标
实现患者信息的**增删改查**四类基础能力，并将数据持久化到 **SQLite** 数据库中。

### 1.3 非目标 (Non-Goals)
- 不涉及 HIS/EMR 级别的复杂病史、医嘱、处方管理
- 不涉及多租户、多医院、权限细分模型
- 不涉及对外开放 API 给第三方系统
- 不涉及患者隐私合规专项（HIPAA/个保法专项评估），但字段设计需预留可扩展空间

## 2. 角色与用户场景

| 角色 | 典型场景 |
| --- | --- |
| 前台/护士 | 新患者登记、查找已有患者、更新联系方式 |
| 医生 | 查看患者基础档案、在就诊前确认患者身份 |
| 管理员 | 修正错误信息、合并重复数据、删除测试数据 |

### 2.1 核心用户故事

1. **US-01 新增患者**：作为前台，我要能录入新患者的基本信息并保存，以便为其建立电子档案。
2. **US-02 查询患者**：作为前台，我要能通过姓名/手机号/证件号快速检索患者，以便复用已有档案，避免重复建档。
3. **US-03 分页浏览**：作为管理员，我要能分页浏览全部患者列表，支持按性别、年龄段、建档时间筛选。
4. **US-04 修改患者**：作为前台，我要能修改患者的联系方式、地址等信息，以便保持数据准确。
5. **US-05 删除患者**：作为管理员，我要能删除错误录入或测试患者数据（软删除优先）。
6. **US-06 查看详情**：作为医生，我要能查看单个患者的完整详情，以便判断就诊背景。

## 3. 功能范围 (Scope)

### 3.1 功能清单

| 编号 | 功能 | 优先级 | 对应 US |
| --- | --- | --- | --- |
| F-01 | 患者信息新增 | P0 | US-01 |
| F-02 | 患者信息查询（分页、筛选） | P0 | US-02, US-03 |
| F-03 | 患者信息修改 | P0 | US-04 |
| F-04 | 患者信息删除 | P0 | US-05 |
| F-05 | 患者信息详情 | P0 | US-06 |

### 3.2 功能详述

#### F-01 新增患者
- 触发：点击「新增患者」按钮，填写表单提交。
- 必填字段：姓名、性别、出生日期、手机号、证件类型、证件号。
- 校验规则：
  - 手机号：大陆 11 位（`^1[3-9]\d{9}$`），或可选填留空。
  - 证件号：身份证需校验 18 位格式与校验位；其它证件类型仅校验非空与长度（≤ 32）。
  - 证件号 + 证件类型 组合需**唯一**，重复时返回明确错误。
  - 出生日期：不晚于今天。
- 成功：返回新建患者 `id`，跳转详情或回到列表。
- 失败：表单内联提示对应字段错误。

#### F-02 查询患者（列表 / 分页 / 筛选）
- 默认展示：按 `created_at` 倒序、每页 20 条。
- 分页参数：`page`（从 1 起）、`page_size`（默认 20，最大 100）。
- 支持筛选：
  - 关键字 `keyword`：对姓名、手机号、证件号做模糊匹配（`LIKE '%kw%'`）。
  - 性别 `gender`：`male` / `female` / `other`。
  - 建档时间范围 `created_from` / `created_to`。
- 响应：`{ total, page, page_size, items[] }`。
- 列表列：姓名、性别、年龄（由出生日期计算）、手机号（脱敏）、证件号（脱敏）、建档时间、操作（查看/编辑/删除）。
- 脱敏规则（列表及默认详情展示）：
  - 手机号：保留前 3 位与后 2 位，中间用 `****` 占位，例如 `138****88`。
  - 证件号（身份证 18 位）：保留前 3 位与后 2 位，中间用 `*` 占位。
  - 证件号（其它类型）：无论长度一律以前 2 位 + `***` + 末 1 位展示，确保不超过 1/3 原文被暴露。
  - 生成脱敏串时必须在后端进行（前端仅展示），避免明文进入浏览器端缓存/日志。

#### F-03 修改患者
- 入口：列表行「编辑」或详情页「编辑」按钮。
- 可修改字段：除 `id`、`created_at` 外的所有字段。
- 证件类型 + 证件号仍需保持全局唯一。
- 保存后刷新详情并提示成功。

#### F-04 删除患者
- 采用**软删除**：在数据库中保留记录，置 `deleted_at` 时间戳；列表/详情查询默认过滤掉软删除数据。
- 删除前需二次确认弹窗。
- 被软删除的患者，其证件号唯一性约束**不再**占用（允许新记录复用相同证件号）。

#### F-05 患者详情
- 展示所有字段原文（可切换"脱敏/明文"查看，默认脱敏）。
- 展示创建时间、最近修改时间。
- 提供「编辑」「删除」快捷操作按钮。

## 4. 数据字段定义（面向 UI / 产品层）

> 落库列名、索引等由 `role:architect` 在架构文档中最终定义，此处仅描述业务语义。

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| id | 整数 | 系统生成 | 主键 |
| name | 字符串(64) | 是 | 姓名 |
| gender | 枚举 | 是 | `male` / `female` / `other` |
| birth_date | 日期 | 是 | 出生日期（YYYY-MM-DD） |
| phone | 字符串(20) | 否 | 手机号 |
| id_type | 枚举 | 是 | `id_card`/`passport`/`other` |
| id_number | 字符串(32) | 是 | 证件号 |
| address | 字符串(255) | 否 | 联系地址 |
| remark | 文本 | 否 | 备注 |
| created_at | 时间戳 | 系统生成 | 建档时间 |
| updated_at | 时间戳 | 系统生成 | 最近修改时间 |
| deleted_at | 时间戳 | 系统生成 | 软删除标记，NULL 表示未删除 |

派生字段：
- **age**：由 `birth_date` 计算当前年龄（前端或后端计算均可）。

## 5. 非功能性需求

| 类别 | 要求 |
| --- | --- |
| 持久化 | 使用 **SQLite** 作为数据库；建库脚本/迁移放在 backend 产物中。 |
| 性能 | 10 万条患者数据下，列表分页查询 < 300ms（本地 SSD）。 |
| 可用性 | 单机部署，无高可用要求；备份通过复制 `.db` 文件实现。 |
| 安全 | 证件号、手机号默认按 §3.2 F-02 的脱敏规则展示（中间位 `*`，只暴露极少量首尾位）；仅在详情页且用户显式点击"查看明文"时显示完整信息，并需记录一次访问日志。 |
| 国际化 | 界面文本采用中文；字段 key 使用英文以便后续扩展。 |
| 日志 | 关键写操作（新增、修改、删除）需记录操作日志（至少 INFO 级别）。 |
| 兼容性 | 后端需跑在主流 Linux/macOS/Windows 上；前端支持最新两个版本的 Chrome/Edge。 |

## 6. 错误处理与边界情况

- 证件号重复：返回业务错误码，前端在对应字段下红字提示"该证件号已存在"。
- 删除不存在或已软删除的 id：返回 404。
- 修改时并发冲突（非必须，v1 不处理乐观锁）：后写覆盖先写。
- 分页参数越界：`page` < 1 按 1 处理；`page_size` > 100 按 100 处理。
- 空结果列表：前端展示"暂无数据"占位图。

## 7. 验收标准 (Acceptance Criteria)

- [ ] AC-01：能通过 UI 新增一个合法患者并在列表中看到。
- [ ] AC-02：重复证件号新增被拒绝，并有明确错误提示。
- [ ] AC-03：列表支持关键字（姓名/手机号/证件号）模糊检索，并支持性别与建档时间筛选。
- [ ] AC-04：分页正确，total / page / page_size 与数据一致。
- [ ] AC-05：编辑任一字段并保存，详情页立即展示最新值。
- [ ] AC-06：删除患者后列表不再展示该条；数据库记录仍在且 `deleted_at` 被填充。
- [ ] AC-07：详情页默认对证件号、手机号脱敏；切换"查看明文"后展示完整值。
- [ ] AC-08：所有数据落入 SQLite 文件，重启服务后数据仍在。
